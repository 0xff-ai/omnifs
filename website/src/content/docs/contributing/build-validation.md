omnifs has two build targets in one workspace: host crates (`omnifs-cli`,
`omnifs-host`) build natively, and providers and tools build directly as
`wasm32-wasip2` WASM components. Validation differs accordingly.

## The Rust baseline

Run this whenever you change host or CLI code:

```bash
cargo fmt
cargo nextest run
```

`cargo fmt` enforces formatting; `cargo nextest run` is the native test runner.
This is the minimum bar for a native-only change.

## Repo checks

When you touch broad surfaces or provider code, run the `just` recipes. They are
thin wrappers over Cargo, Bun, Docker, and shell - the root `justfile` is the
human command surface (run `just` to list grouped commands).

```bash
just check             # fmt-check; host clippy/tests; provider checks; npm + scripts validate
just providers-check   # wasm32-wasip2 check/clippy/test for providers and tools
just providers-build   # release-build providers and tools for wasm32-wasip2
```

`just check` runs the full gate in sequence: `fmt-check`, `host-clippy`,
`test` (which builds provider WASM then runs host tests), `providers-check`,
`npm-validate`, and `scripts-test`. The host clippy/test lanes **exclude** the
WASM packages, then `providers-check` runs them under the WASM target:

```bash
cargo clippy --workspace \
  --exclude 'omnifs-provider-*' --exclude 'omnifs-tool-*' --exclude test-provider \
  --all-targets -- -D warnings
```

Package selection always uses cargo `-p` / `--exclude` globs
(`omnifs-provider-*`, `omnifs-tool-*`, `test-provider`), never a hand-maintained
crate list.

## Provider build, clippy, and test

Providers build directly as components with the Rust `wasm32-wasip2` target.
`cargo build --target wasm32-wasip2` emits the provider component artifacts
directly - there is no separate component-tooling step.

Provider clippy and test commands must include `--target wasm32-wasip2` and the
`-p` globs:

```bash
cargo clippy -p 'omnifs-provider-*' -p 'omnifs-tool-*' -p test-provider --target wasm32-wasip2 -- -D warnings
cargo test -p 'omnifs-provider-*' -p test-provider --target wasm32-wasip2 --no-run
```

:::caution
WASM tests compile but cannot execute on the host - there is no WASM runtime in
the test harness. Use `--no-run` for target-specific provider compilation
checks. Tests that should actually run must use `#[cfg(test)]` with
host-target-compatible code.
:::

## WIT bindings

WIT bindings are generated by `wit_bindgen::generate!` inside each provider's
`lib.rs`, against the shared contract under `wit/`. There is no checked-in
generated-bindings file to regenerate; changing the WIT and rebuilding is enough.

## When to use each

| Change | Run |
|---|---|
| Host/CLI code only | `cargo fmt` + `cargo nextest run` |
| Broad surfaces, or any provider code | `just check` |
| Provider source, fast feedback | `just providers-check` |
| Need provider WASM artifacts | `just providers-build` |
| Mount/provider/clone/traversal/runtime behavior | the above **plus** the live runtime path (`omnifs dev -y`, `omnifs status`, smoke demo, `tail /tmp/omnifs.log`) - see [testing](/contributing/testing/) |

:::note
CI builds Rust artifacts natively and uses Docker only to assemble the runtime
image. Linux CLI artifacts use `cargo-zigbuild` with a GNU glibc 2.17 baseline;
Darwin CLI artifacts are cross-linked from Linux. Provider and tool WASM
artifacts are built by `just providers-build` with WASI SDK pins from
`tools/versions.toml`. The CI lanes wrap these same recipes (`ci-host`,
`ci-wasm`).
:::
