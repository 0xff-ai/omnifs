# Mount lifecycle

Status: core model aligned; CLI surface table partially ahead of the shipped commands
Scope: `crates/omnifs-cli`, `crates/omnifs-host/src/auth.rs`, `omnifs_mount::mounts` (mount config), `crates/omnifs-home/src/lib.rs` (home layout), provider metadata, Docker lifecycle wrappers
Related: `docs/design/host-auth.md`, `docs/design/file-attributes.md`, `docs/design/wasm-sandbox-substrate.md`

## Context

omnifs projects configured providers into a Linux FUSE mount. Historically the working configuration lived as enabled provider JSON files under `docker/providers/` and credentials were staged by hand. Provider-owned defaults now live under `providers/<name>/`, are embedded into the provider wasm as `omnifs.provider-metadata.v1`, and are mirrored into the host CLI's built-in catalog for the bundled providers.

That shape mixed three concepts:

- provider metadata, which is shipped by the project and describes what a provider can do
- enabled mounts, which are user or developer choices about what should be mounted
- runtime credentials, which live in the host `credentials.json` file consumed by the trusted daemon

The accepted implementation separates those concepts. Provider metadata is shipped in the provider artifact and in the CLI catalog for first-run host workflows. Enabled mounts are explicit, thin configuration generated by `omnifs init`. Credentials are stored through the host file store and exposed to the trusted daemon through the writable `OMNIFS_HOME` bind.

## Design model

A provider metadata section describes a possible mount. It declares the provider component, default mount name, credential shape, validation hints, and non-secret provider defaults. It is not itself an enabled mount.

An enabled mount is a concrete instance. It has a mount name, provider component, and the chosen auth scheme or account. Runtime capability grants and default provider config are inherited from the built-in catalog on the host and from wasm metadata in the runtime unless the mount explicitly overrides them.

A runtime session is the running container plus any explicit host binds needed by configured providers. Credentials are not copied into session files; the container writes back to the host `credentials.json` directly.

This gives the implementation a simple ownership boundary:

- shipped provider metadata says what can be configured
- user mount config says what is enabled for normal `omnifs up`
- dev mount configs live in `providers/<name>/dev-mount.json` when dev wiring differs from `{provider, mount}` defaults; `build.rs` embeds them into the CLI, and `omnifs dev` pushes them to the daemon over the control API like any other mount
- the file-backed credential store is durable and shared with the trusted daemon
- explicit provider preopens remain runtime-only container binds

## Two-commit shape

The work should stay packaged as two logical commits.

First commit: record this design and the accepted boundaries.

Second commit: implement the metadata/config separation and the lifecycle behavior around the file-backed credential store. This includes validating credentials for `omnifs up` and binding the host `OMNIFS_HOME` tree as the daemon's writable runtime home.

The split matters because the first commit is the contract and the second commit is runtime behavior. Keeping them separate makes review and rollback possible.

## Mount scopes

`omnifs up` defaults to user mounts. A normal user running `omnifs up` should not implicitly enable repo fixtures or development mounts.

Repo and dev mounts are explicit. They are used for reproducible demos, smoke tests, and development workflows, but they must be selected by the command path or flag that is clearly development-oriented.

Expected behavior:

| Flow | Enabled mounts |
|---|---|
| `omnifs up` | user mounts |
| repo demo or CI path | repo mounts, explicitly selected |
| dev path | dev mounts and source bind mounts, explicitly selected |

This avoids surprising users with mounts they did not configure while preserving the current container workflow for development.

## Provider metadata and templates

Provider metadata is shipped by the provider wasm and mirrored in the CLI's built-in catalog. It describes the provider, credential requirements, validation hints, default auth scheme, and default non-secret config. The provider-owned source is `providers/<name>/omnifs.provider.json`; its `auth` block is what the CLI uses for host-side login, and what the provider build embeds into the wasm `omnifs.provider-metadata.v1` section.

Enabled mount config is separate. It records the user's chosen mount name, provider component, and chosen auth scheme. It may override non-secret settings, but it must not be treated as the source of provider capability metadata.

The important invariant is one-way use: metadata/templates help create mounts, but a shipped template does not become enabled just because it exists in the repo.

## Credential store

The host owns durable credentials, and the trusted container receives the host `OMNIFS_HOME` tree as its writable runtime home.

The accepted store shape is deliberately single-backend:

- production: file store under the user's omnifs config directory, mode 600
- test: memory store

Reads and writes use the same resolved `credentials.json` file across local builds, installed binaries, CI, containers, and the runtime auth refresh path. This avoids interactive credential backend prompts during `omnifs up` and related startup flows.

The file store is a pragmatic local credential store, not a stronger security boundary. Encryption-at-rest belongs at the OS or disk layer for this design. The daemon reads and refreshes this store directly through the trusted runtime-home bind.

## Runtime credential access

Before starting the container, the host CLI resolves enabled mounts and verifies that any host-managed credentials exist in the file store. It then starts the container with the host omnifs home mounted writable at the daemon's `OMNIFS_HOME`.

Mount configs presented to the daemon keep host-managed auth as `scheme` plus optional `account`. Explicit `token_file` and `token_env` remain available for user-managed external secrets.

Credential files must not be written under the repo working tree. `omnifs dev` provisions contributor credentials into a dedicated dev home (`~/.omnifs/dev`) via the same store path as `omnifs init`, never into the checkout.

## CLI surface

The implemented surface should only claim commands that exist. The current shipped lifecycle surface is:

| Command | Phase | Purpose |
|---|---|---|
| `omnifs up` | shipped | Start the container from enabled user mounts by default. |
| `omnifs dev` | shipped | Start the contributor sandbox container with explicit dev mounts. |
| `omnifs down` | shipped | Stop the container. |
| `omnifs status` | shipped | Report runtime and config state. |
| `omnifs setup` | shipped | Guided onboarding walkthrough; runs `init` per chosen provider, then launches the container. |
| `omnifs init` | shipped | Create an enabled mount from provider metadata and run default auth. |
| `omnifs mounts rm <name>` | shipped | Remove an enabled mount and (by default) its stored credential. |
| `omnifs doctor` | shipped | Diagnose environment and auth. |
| `omnifs reset` | shipped | Remove every mount config and (by default) its stored credential, then stop the container. |

Design prose and help text should not present phased commands as shipped behavior. When a command is not implemented, describe it as planned or leave it out of user-facing claims.

## `omnifs up`

`omnifs up` is the normal user entrypoint. It should:

1. Resolve user enabled mounts.
2. Resolve each mount's provider metadata or template reference.
3. Validate credentials through the file store when needed.
4. Start the container with the selected mount config and writable runtime home.
5. Push mount specs to the daemon.

The default command should not include repo fixtures or dev-only mounts. Those paths are selected explicitly by development and demo commands.

## `omnifs down`

`omnifs down` stops the running container. Credentials are durable host state and are not cleaned up by lifecycle commands.

## Headless and development paths

Headless and CI flows continue to use file or environment backed credentials. The daemon reads host-managed credentials from `credentials.json` and still accepts explicit `token_file` / `token_env` external sources.

Repo and dev mounts remain useful for:

- reproducible smoke tests
- container demos
- local development with source bind mounts

They are not enabled by ordinary `omnifs up` unless explicitly selected.

## Phased future

The later interactive lifecycle is still valid as a design direction, but it is not part of the shipped claim until implemented.

Current shipped flow:

1. `omnifs init github` writes a `[[mounts]]` entry in `~/.omnifs/config.toml` and runs the GitHub device flow.
2. `omnifs init linear` writes a `[[mounts]]` entry in `~/.omnifs/config.toml` and runs the Linear OAuth flow.
3. `omnifs up` materializes the selected credentials and starts the container.

Future docs should keep this distinction clear: `init` owns mount generation for normal users; provider-owned defaults live in the CLI catalog and wasm metadata, both derived from `providers/<name>/omnifs.provider.json`; `omnifs dev` is the contributor sandbox path, synthesizing mount configs from the built-in provider manifests and launching the dev container directly.

## Non-goals

Do not reintroduce macOS-specific mount behavior, `diskutil`, or macFUSE assumptions. The supported runtimes are Linux FUSE (the container workflow) and the read-only NFSv4 loopback frontend (`omnifsd --frontend nfs`), which mounts host-native with no kernel extension.

Do not make the daemon a credential-store client. The daemon consumes files supplied to the session.

Do not treat shipped provider metadata as an enabled mount.

Do not claim provider installation, live validation, credential revocation, or non-interactive init support as shipped until those commands exist.
