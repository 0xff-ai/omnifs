# Mount lifecycle

Status: implementation aligned
Scope: `crates/omnifs-cli`, `crates/omnifs-host/src/auth.rs`, `crates/omnifs-host/src/config`, provider metadata, Docker lifecycle wrappers
Related: `docs/design/host-auth.md`, `docs/design/file-attributes.md`, `docs/design/wasm-sandbox-substrate.md`

## Context

omnifs projects configured providers into a Linux FUSE mount. Historically the working configuration lived as enabled provider JSON files under `docker/providers/` and credentials were staged by hand. Provider-owned defaults now live under `providers/<name>/`, are embedded into the provider wasm as `omnifs.provider-metadata.v1`, and are mirrored into the host CLI's built-in catalog for the bundled providers.

That shape mixed three concepts:

- provider metadata, which is shipped by the project and describes what a provider can do
- enabled mounts, which are user or developer choices about what should be mounted
- runtime credential files, which are session artifacts consumed by the container

The accepted implementation separates those concepts. Provider metadata is shipped in the provider artifact and in the CLI catalog for first-run host workflows. Enabled mounts are explicit, thin configuration generated by `omnifs init`. Credentials are stored through a layered host store and materialized into per-session files only for `omnifs up`.

## Design model

A provider metadata section describes a possible mount. It declares the provider component, default mount name, credential shape, validation hints, and non-secret provider defaults. It is not itself an enabled mount.

An enabled mount is a concrete instance. It has a mount name, provider component, and the chosen auth scheme or account. Runtime capability grants and default provider config are inherited from the built-in catalog on the host and from wasm metadata in the runtime unless the mount explicitly overrides them.

A session is the running container plus its temporary credential directory. Session files are disposable. If `up` or `down` fails partway through, the CLI must clean any session directory it created before returning the error.

This gives the implementation a simple ownership boundary:

- shipped provider metadata says what can be configured
- user mount config says what is enabled for normal `omnifs up`
- dev mount configs live in `providers/<name>/dev-mount.json` when dev wiring differs from `{provider, mount}` defaults; `build.rs` embeds them into the CLI and container entrypoint materializes them at `/root/.omnifs/mounts/`
- the credential store is durable
- the session credential directory is temporary

## Two-commit shape

The work should stay packaged as two logical commits.

First commit: record this design and the accepted boundaries.

Second commit: implement the metadata/config separation and the lifecycle behavior around credentials and sessions. This includes the layered credential store, materializing credentials for `omnifs up`, and cleanup on failed `up` or `down`.

The split matters because the first commit is the contract and the second commit is runtime behavior. Keeping them separate makes review and rollback possible.

## Mount scopes

`omnifs up` defaults to user mounts. A normal user running `omnifs up` should not implicitly enable repo fixtures or development mounts.

Repo and dev mounts are explicit. They are used for reproducible demos, smoke tests, and development workflows, but they must be selected by the command path or flag that is clearly development-oriented.

Expected behavior:

| Flow | Enabled mounts |
|---|---|
| `omnifs up` | user mounts |
| repo demo or CI path | repo mounts, explicitly selected |
| dev path | dev mounts and source bind mounts, explicitly selected |

This avoids surprising users with mounts they did not configure while preserving the current container workflow for development.

## Provider metadata and templates

Provider metadata is shipped by the provider wasm and mirrored in the CLI's built-in catalog. It describes the provider, credential requirements, validation hints, default auth scheme, and default non-secret config. The provider-owned source is `providers/<name>/omnifs.provider.json`; its `auth` block is what the CLI uses for host-side login, and what the provider build embeds into the wasm `omnifs.provider-metadata.v1` section.

Enabled mount config is separate. It records the user's chosen mount name, provider component, and chosen auth scheme. It may override non-secret settings, but it must not be treated as the source of provider capability metadata.

The important invariant is one-way use: metadata/templates help create mounts, but a shipped template does not become enabled just because it exists in the repo.

## Credential store

The host owns durable credentials. The container only receives session files.

The accepted store shape is layered:

- primary: OS keychain through the host CLI
- fallback: file store under the user's omnifs config directory, mode 600
- test: memory store

Reads check the primary store first, then the fallback. Writes prefer the primary store when available. If the keychain is unavailable, the CLI can fall back to the file store with an explicit warning.

The file store is a pragmatic fallback, not a stronger security boundary. Encryption-at-rest belongs at the OS or disk layer for this design. The daemon does not read the keychain or the fallback store directly.

## Runtime credential materialization

Before starting the container, the host CLI resolves enabled mounts and fetches any needed credentials from the layered store. It writes those credentials into a newly-created per-session directory with restrictive permissions, then bind-mounts that directory into the container read-only.

Mount configs presented to the daemon use the existing file-based credential surface, such as `token_file`. The daemon remains credential-store blind.

The session directory is owned by the lifecycle command that created it. Cleanup rules:

- if `omnifs up` fails after creating a session directory, remove that directory before returning
- if `omnifs down` fails partway through, still attempt to remove the session directory
- stale session directories from earlier failed runs should be eligible for cleanup by the next lifecycle command

Credential files must not be written under the repo working tree.

## CLI surface

The implemented surface should only claim commands that exist. The long-term lifecycle shape is:

| Command | Phase | Purpose |
|---|---|---|
| `omnifs up` | current lifecycle work | Start the container from enabled user mounts by default. |
| dev or repo up path | current lifecycle work | Start the container with explicit repo or dev mounts. |
| `omnifs down` | current lifecycle work | Stop the container and clean session credentials. |
| `omnifs status` | existing or phased | Report runtime and config state. Do not claim live validation unless implemented. |
| `omnifs init` | current lifecycle work | Create an enabled mount from provider metadata and run default auth. |
| `omnifs list` | phased | List enabled mounts. |
| `omnifs configure` | phased | Edit an existing enabled mount. |
| `omnifs rotate` | phased | Replace a stored credential. |
| `omnifs doctor` | phased | Run live provider validation. |
| `omnifs remove` | phased | Remove an enabled mount and optional credential. |

Design prose and help text should not present phased commands as shipped behavior. When a command is not implemented, describe it as planned or leave it out of user-facing claims.

## `omnifs up`

`omnifs up` is the normal user entrypoint. It should:

1. Resolve user enabled mounts.
2. Resolve each mount's provider metadata or template reference.
3. Resolve credentials through the layered store when needed.
4. Create a session credential directory.
5. Materialize credential files for the container.
6. Start the container with the selected mount config and credential directory.
7. Clean the session directory if startup fails.

The default command should not include repo fixtures or dev-only mounts. Those paths are selected explicitly by development and demo commands.

## `omnifs down`

`omnifs down` stops the running container and cleans session credentials. Cleanup is part of the command contract, not best effort hidden in logs.

If stopping the container fails, the command should still attempt session cleanup and report both outcomes. The credential cleanup result should be visible enough that a user can tell whether any runtime secret files remain.

## Headless and development paths

Headless and CI flows continue to use file or environment backed credentials. The daemon should keep accepting the same file-oriented credential surface it already understands.

Repo and dev mounts remain useful for:

- reproducible smoke tests
- container demos
- local development with source bind mounts

They are not enabled by ordinary `omnifs up` unless explicitly selected.

## Phased future

The later interactive lifecycle is still valid as a design direction, but it is not part of the shipped claim until implemented.

Current shipped flow:

1. `omnifs init github` creates `~/.omnifs/config/mounts/github.json` and runs the GitHub device flow.
2. `omnifs init linear` creates `~/.omnifs/config/mounts/linear.json` and runs the Linear OAuth flow.
3. `omnifs up` materializes the selected credentials and starts the container.

Future docs should keep this distinction clear: `init` owns mount generation for normal users; provider-owned defaults live in the CLI catalog and wasm metadata, both derived from `providers/<name>/omnifs.provider.json`; `omnifs dev` is the contributor sandbox path, synthesizing mount configs from the built-in provider manifests and launching the dev container directly.

## Non-goals

Do not reintroduce macOS-specific mount behavior, `diskutil`, or macFUSE assumptions. The supported runtime remains Linux FUSE through the container workflow.

Do not make the daemon a credential-store client. The daemon consumes files supplied to the session.

Do not treat shipped provider metadata as an enabled mount.

Do not claim provider installation, live validation, credential revocation, or non-interactive init support as shipped until those commands exist.
