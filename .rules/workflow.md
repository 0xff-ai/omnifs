# Workflow

**Read when:** starting a session, building, running tests, or running the
project locally. Re-read if a build/test command isn't doing what you expect.

**Update when:** adding or changing a build target, a `just` recipe, the
Docker Compose flow, the provider build pipeline (e.g. reintroducing a
preview1 adapter step), test-harness conventions, or interactive shell
defaults baked into the image.

## Scope

- Linux-only. Don't reintroduce macOS-specific mount behavior, `diskutil`,
  or macFUSE assumptions unless explicitly requested.
- The primary supported workflow is the **container workflow** in
  `compose.yaml`.

## Run the project

```bash
mkdir -p .secrets
gh auth token > .secrets/github_token
docker compose up --build -d            # build, start, mount /omnifs
docker compose exec omnifs /bin/zsh     # interactive shell
docker compose logs omnifs              # container logs
docker compose down                     # stop
```

`docker compose up --build -d` mounts omnifs at `/omnifs` and exposes a
top-level symlink (`/<mount>`) inside the container for every configured
provider in `~/.omnifs/providers/*.json` (currently `/arxiv`, `/dns`,
`/github`).

`just` is a wrapper for the same flow plus build/test recipes (`just dev`,
`just shell`, `just check`, `just test`). Don't add alternate local mount
recipes unless asked.

## Build and test

Host crates (`cli`, `host`, `omnifs-sdk`, `omnifs-sdk-macros`) build for the
native target. Providers build for `wasm32-wasip2`.

```bash
just check          # fmt, clippy, host tests, provider checks
just check-providers  # clippy + test-compile for wasm32-wasip2 providers
cargo fmt
cargo test          # host crates only
```

Provider clippy/test commands must include `--target wasm32-wasip2`:

```bash
cargo clippy -p omnifs-provider-dns --target wasm32-wasip2 -- -D warnings
cargo test   -p omnifs-provider-dns --target wasm32-wasip2 --no-run
```

WASM tests compile but cannot execute on the host (no WASM runtime in
Cargo's test harness). Always use `--no-run` for `wasm32-wasip2` tests.

## Provider build pipeline

Providers are built directly as components with the Rust `wasm32-wasip2`
target. `cargo build --target wasm32-wasip2` emits the component artifact;
WIT bindings are generated by the `wit_bindgen::generate!` macro in each
provider's `lib.rs`.

Do not reintroduce a preview1 adapter step unless the runtime design
explicitly needs to load preview1 guest modules again.

## Validate runtime path for risky changes

For mount, provider, or clone changes, do not stop at Rust-only checks.
Validate through the supported runtime path:

```bash
just dev
docker exec omnifs /bin/zsh -lc 'omnifs status'
docker exec omnifs /bin/zsh -lc 'OMNIFS_DEMO_MODE=smoke /tmp/demo.sh'
docker exec omnifs /bin/zsh -lc 'tail -n 80 /tmp/omnifs.log'
```

## Dockerfile

The Dockerfile is intentionally cache-oriented:

- multi-stage build
- `cargo-chef`
- BuildKit cache mounts
- minimal build context via `.dockerignore`

Preserve that structure unless there is a clear regression or simplification
with equal caching behavior.

## Shell expectations inside the container

Ubuntu 25.10, `zsh`. `ls` is aliased to `ls --color=auto`; `ll` is aliased
to `ls -lrt`. If changing shell behavior, prefer the image rather than
per-session config.
