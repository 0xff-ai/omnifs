## Notes

- **How produced.** `just dev -y --detach` (default profile: arxiv, db, dns,
  github, k8s), then `docker restart omnifs` for cold caches, then the suite ran
  *inside* the container as a compiled Linux binary against the local `k8s`
  fixture (a real k3s cluster). Running in-container avoids the hundreds of
  milliseconds of `docker exec` startup that would otherwise pollute p50/p95;
  each sample is one in-container `Bun.spawn` of the real tool, so it includes
  that tool's own `fork`/`exec` cost (intrinsic to invoking `ls`/`cat`/`grep`).
- **Why k8s.** The `k8s` mount is served by a local fixture, so these numbers
  isolate omnifs's own overhead rather than upstream API latency. `db` was
  degraded this run (missing preopen grant) and `github` had no token, so
  neither was used.
- **Reads within K3.** `ls`, `ls_subdir`, and `cat` hold warm p95 <= 2.7 ms
  through concurrency 8, comfortably under the 50 ms K3 target.
- **grep_r is the real signal.** `grep -r apiVersion` re-walks and re-reads all
  69 files under `apiservices` every iteration. Warm p95 already crosses 50 ms
  at concurrency 4 (52 ms) and reaches 224 ms at concurrency 8, where eight
  concurrent full-subtree scans contend. Thresholds here are recorded, not
  enforced; this is a genuine head-of-line/contention data point for T4 and the
  frontend-concurrency work, not a pass/fail.
- **Cold caveat.** Cold is a fresh-daemon first touch after `docker restart`,
  not upstream-cold: the on-disk cache under `OMNIFS_HOME` survives a restart.
  `ls_subdir` cold (26.9 ms) reflects the first provider callout to populate the
  `apiservices` listing; `grep_r` cold (63.8 ms) reflects the first cold read of
  the whole subtree.
